# AppThrive: Merchant Success Platform for Shopify App Developers
## Developer Milestone Document (Revised)

> **Original:** Generated by ChatGPT. **Revised:** 2026-04-25 after deep codebase + PRD gap analysis.
> **Source of truth:** `docs/AppThrive-PRD-v1.3.md`. This document aligns milestones to the PRD and reflects what has actually been built.

### Primary Goal
Build a multi-tenant merchant success platform for Shopify app developers to manage merchant lifecycle, intelligence scoring, lifecycle automation, communication, and retention — using BYO providers (email, AI, messaging).

---

## 1. Core Data Ownership Rules

### 1.1 Tenant Model
- Multi-tenant platform (Clerk orgs)
- Each organization = workspace
- Workspace can own multiple Shopify apps
- All data belongs to exactly one workspace
- Per-org DEK envelope encryption (AES-256-GCM) for PII/secrets

### 1.2 Store Ownership Model
- Same Shopify store can exist in multiple workspaces
- Records must remain isolated per workspace
- No cross-workspace data access, merge, or visibility

### 1.3 App-Scoped Operational Model
Within a workspace:
- One store → multiple app installations
- Each app has independent: plan, trial, events, lifecycle state, metrics
- Cross-app aggregation allowed ONLY within the same workspace

### 1.4 Security Rule
Ownership resolution order:
1. Workspace (orgId on every query)
2. App (appId scoping for operational data)
3. Role/Permission (Clerk RBAC: viewer, editor, admin)

## M0 — Tenant Isolation & Auth ✅ COMPLETE

### What's Built
- Clerk auth with org-scoped sessions, invite-gated signup
- RBAC roles: viewer, editor, admin (via Clerk)
- `orgId` on ALL database queries (enforced in server actions)
- Per-org DEK envelope encryption (AES-256-GCM, Phase 22)
- Audit logging framework (`@appthrive/audit`)
- API key management (PATs with scoped auth)
- Admin panel with OAuth + MFA (Auth.js v5, TOTP mandatory)

### Acceptance ✅
- No cross-tenant access possible
- All data tied to workspace via orgId foreign keys
- Secrets encrypted at rest with per-org keys

## M3 — Event & Webhook Pipeline ✅ 95% COMPLETE

### What's Built
- 3-path ingest: SDK `track()`/`incrementMetric()`, developer webhook POST, Shopify-direct GDPR
- HMAC-SHA256 signature verification with 5-min timestamp window
- Idempotency via `idempotencyKey` (Upstash SETNX)
- Metric rollup pipeline: hourly/daily/weekly/monthly aggregation
- Outbound webhooks: 10 event types, signed delivery, retry ladder (6 attempts), dead-letter tracking
- Webhook secret rotation
- Batch event ingestion (100 events per call)

### Gaps
- **Event timeline UI** — no user-facing event stream on merchant detail page (PRD §5.6 "Timeline tab")
- **Event replay/forensics** — no ability to replay failed events or view full event payload in UI
- **Dead letter queue UI** — DLQ entries tracked in DB but no admin surface to review/retry

## 5. Engineering Rule

> All merchant data is workspace-owned.
> All operational data is app-scoped.
> No cross-workspace exposure is allowed.
> All external input validated with Zod.
> All PII encrypted at rest with per-org keys.
> Money stored as bigint cents in USD.
> Timestamps stored as UTC, displayed in user's timezone.
